# Oratio - A Writing Style Guide MCP Server

An MCP server that turns your corporate Writing Style Guide PDF into an AI expert you can query from **Claude Desktop**, **VS Code**, or directly in [claude.ai](https://claude.ai). Supports **Claude**, **GitHub Copilot**, **OpenRouter**, and **Ollama** (fully local) as LLM backends — switchable with a single line in `.env`.

Works on **macOS**, **Ubuntu**, and **Windows via WSL2**.

---

## How It Works

The project has three separate modes of use. Each builds on the same RAG engine but exposes it differently.

### Mode 1 — MCP Server (Claude Desktop / VS Code Chat)

```
Your question
      │
      ▼
Claude Desktop or VS Code Copilot Chat
      │  MCP protocol
      ▼
server.py  (MCP server)
      │
      ▼
rag.py  (RAG engine)
  ├── PyPDF  →  loads and chunks the PDF
  ├── FAISS  →  finds relevant chunks via semantic search
  └── LLM    →  Claude / Copilot / OpenRouter / Ollama
      │
      ▼
Answer + source page numbers
```

### Mode 2 — VS Code Extension (right-click style checking)

Two processes must run simultaneously:

```
VS Code (right-click → Check Selection)
      │  HTTP POST /check
      ▼
server_http.py  (lightweight HTTP server, port 5123)
      │
      ▼
rag.py  (RAG engine — same as above)
      │
      ▼
LLM  (whichever backend is set in .env)
      │
      ▼
Results shown in Output panel + squiggles in editor
```

So you need two terminals running:
- **Terminal 1:** `ollama serve` (if using Ollama — can be a background service instead)
- **Terminal 2:** `make server-http`

### Mode 3 — Claude.ai Project (no server required)

```
make skill
      │
      ▼
generate_skill.py
  ├── Extracts text from PDF
  └── LLM generates a system prompt summarising all rules
      │
      ▼
skill_output/system_prompt.md  ← paste into Claude.ai Project Instructions
skill_output/INSTRUCTIONS.md   ← setup guide

Then in claude.ai:
  Project Instructions (system prompt) + PDF uploaded as knowledge
      │
      ▼
Claude answers style questions natively — no server needed
```

The first time a query runs in Mode 1 or 2, the PDF is loaded, chunked, and embedded into a FAISS vector store in memory. Subsequent queries in the same session are fast because the store is reused.

## Privacy & Data Security

**Short answer:** your PDF never leaves your machine as a file. However, small excerpts of its text are sent to your chosen LLM backend as part of each query prompt.

### What stays on your machine — always

| Component | Where it runs |
|---|---|
| PDF file | Local disk only — never uploaded |
| Text chunking | Local Python process |
| Embeddings model (`all-MiniLM-L6-v2`) | Runs fully locally via `sentence-transformers` |
| FAISS vector store | In-memory only, rebuilt each session |

### What leaves your machine

When you ask a question or check text, the RAG engine retrieves the most relevant PDF chunks (~5–8 passages of ~1000 characters each) and sends them to the LLM alongside your query. These are plain text excerpts, not the PDF itself.

| Backend | Destination | Notes |
|---|---|---|
| **Ollama** | `localhost` only — fully offline | Nothing leaves your machine. Best for confidential documents. |
| **Claude (Anthropic)** | `api.anthropic.com` | Anthropic does not train on API data by default. See [anthropic.com/privacy](https://www.anthropic.com/privacy) |
| **GitHub Copilot** | Microsoft / Azure OpenAI | Subject to your GitHub or enterprise agreement |
| **OpenRouter** | `openrouter.ai` → third-party model host | Requests routed to external providers. See [openrouter.ai/privacy](https://openrouter.ai/privacy) |

### Recommendations by sensitivity level

**Public or internal-only style guide** — any backend is fine.

**Confidential style guide** (unreleased product names, legal language, restricted terminology):
- Prefer **Ollama** — nothing leaves your machine.
- If using cloud, prefer **Claude API** or **GitHub Copilot** under an enterprise agreement.
- Avoid **OpenRouter free tier** for sensitive content.
- Consider reducing `RETRIEVER_K` in `.env` (e.g. to `3`) to limit excerpts sent per query.

**Highly restricted** (cannot leave corporate network):
- Use **Ollama** — see [Running Fully Offline with Ollama](#running-fully-offline-with-ollama).

## Setup

### Prerequisites

**macOS:**
```bash
brew install python@3.13
xcode-select --install   # provides make
```

**Ubuntu / WSL2:**
```bash
sudo apt update && sudo apt install -y python3 python3-pip python3-venv make
```

> **WSL2 note:** run all commands below inside the WSL terminal. Keep project files inside the WSL filesystem (`/home/...`) — not on `/mnt/c/...` — to avoid slow cross-filesystem I/O.

---

### 1. Get the project

```bash
git clone <your-repo-url> mcp-styleguide
cd mcp-styleguide
```

### 2. Install dependencies

```bash
make install
```

This creates `.venv/` using the best available Python (3.13 → 3.12 → 3) and installs all dependencies. Override if needed:

```bash
make install PYTHON=python3.12
```

### 3. Configure `.env`

```bash
cp .env.example .env
```

Key variables:

| Variable | Description |
|---|---|
| `LLM_BACKEND` | `claude`, `copilot`, `openrouter`, or `ollama` |
| `ANTHROPIC_API_KEY` | From [console.anthropic.com](https://console.anthropic.com) |
| `ANTHROPIC_BASE_URL` | Keep as `https://api.anthropic.com` — prevents local proxy conflicts |
| `GITHUB_TOKEN` | From [github.com/settings/tokens](https://github.com/settings/tokens) — needs **Models (read)** scope |
| `OPENROUTER_API_KEY` | From [openrouter.ai/keys](https://openrouter.ai/keys) |
| `OLLAMA_MODEL` | e.g. `mistral:7b` |
| `PDF_PATH` | Absolute path to your style guide PDF |

### 4. Add your PDF

Set `PDF_PATH` in `.env` to the absolute path of your PDF:

```bash
# macOS
PDF_PATH=<PATH>/style-guide.pdf

# Ubuntu / WSL2
PDF_PATH=<PATH>/style-guide.pdf
```

### 5. Test

```bash
make test
```

You should see the PDF load, chunks being created, embeddings generated, and a test answer printed.

## Connecting to Claude Desktop

Open Claude Desktop → **Settings** → **Developer** → **Edit Config** and add:

**macOS:**
```json
{
  "mcpServers": {
    "writing-style-guide": {
      "command": "<PATH>/mcp-styleguide/.venv/bin/python",
      "args": ["<PATH>/mcp-styleguide/server.py"]
    }
  }
}
```

**Ubuntu / WSL2** (Claude Desktop runs on Windows, server runs in WSL):
```json
{
  "mcpServers": {
    "writing-style-guide": {
      "command": "wsl.exe",
      "args": ["--", "<WSL_PATH>/mcp-styleguide/.venv/bin/python",
               "<WSL_PATH>/mcp-styleguide/server.py"]
    }
  }
}
```

Save and restart Claude Desktop. You should see a **Connectors** menu in the chat (under '+' sign), and **writing-style-guide** in there.

**Example prompts:**
- *"Can I use 'can't' in documentation? Check the writing style rules."*
- *"How should I format IP addresses according to our style guide?"*
- *"Is the term 'master/slave' approved in our docs?"*

### Option B — VS Code Extension

This gives you right-click style checking directly in the editor. Results appear in the **Output panel** and as squiggly underlines on the offending text.

#### How it works (communication flow)

```
VS Code editor
  └─ right-click → "Style Guide: Check Selection"
        │  HTTP POST /check  { "text": "..." }
        ▼
server_http.py  running on localhost:5123
        │
        ▼
rag.py  (loads PDF, searches FAISS, calls LLM)
        │
        ▼
LLM  (Ollama / Claude / OpenRouter / Copilot)
        │
        ▼
JSON response  { verdict, issues[], summary }
        │
        ▼
VS Code  →  Output panel (full report)
         →  Squiggles on offending text
         →  Status bar  ✗ 1 fail, 2 warn
```

#### Step 1 — Start the HTTP server

```bash
# In a dedicated terminal — keep it running while you work
make server-http
```

Expected output:
```
Style Guide HTTP server starting on http://localhost:5123
Endpoints: POST /check  POST /ask  GET /health
```

If using Ollama, make sure it is also running (`ollama serve` or as a system service) before starting the HTTP server.

#### Step 2 — Build and install the extension

> **Ubuntu prerequisite:** `sudo apt install -y nodejs npm`
> **MacOS prerequisite:** `brew install node`

```bash
cd vscode
npm install && npm run compile
npm run package    # compiles TypeScript and produces style-guide-checker-0.2.0.vsix
```

Install in VS Code (recommended)
```
Cmd+Shift+P / Ctrl+Shift+P  →  Extensions: Install from VSIX  →  select the .vsix file
```

Or for development (no install, reloads on recompile):
```bash
npm install && npm run compile
# Open the vscode/ folder in VS Code and press F5
```

#### Step 3 — Use it

**Right-click menu** (select text first to see "Check Selection"):

| Action | When available | What it does |
|---|---|---|
| **Style Guide: Check Selection** | Text is selected | Checks only the selected text |
| **Style Guide: Check Full Document** | Always | Checks the entire open file |
| **Style Guide: Ask a Question** | Always | Opens input box, sends question to `/ask` |

**Status bar** (bottom right): click `$(book) Style Guide` at any time to check the full document.

**Results:**
- **Output panel** ("Style Guide" channel) opens automatically showing the full report: verdict, each issue with the offending excerpt and suggested fix, page references, and a summary
- **Squiggles** appear on the exact words that violate rules — red for `FAIL`, yellow for `WARN`
- **Hover** over a squiggle to see the rule and fix inline

**Command palette** (`Cmd+Shift+P` / `Ctrl+Shift+P`):
- `Style Guide: Check Selection`
- `Style Guide: Check Full Document`
- `Style Guide: Ask a Question`
- `Style Guide: Clear All Diagnostics`

#### Extension settings

Search for `styleGuide` in VS Code settings (`Cmd+,` / `Ctrl+,`):

| Setting | Default | Description |
|---|---|---|
| `styleGuide.serverUrl` | `http://localhost:5123` | URL of the HTTP server |
| `styleGuide.timeoutMs` | `180000` | Request timeout in ms. Increase for slow Ollama models. |

#### Troubleshooting the extension

**Status bar shows `Server Offline`**
→ Run `make server-http` in a terminal and keep it open.

**Right-click menu doesn't show Style Guide items**
→ Reload VS Code (`Cmd+Shift+P` → **Developer: Reload Window**) after installing the VSIX.

**Check Selection is greyed out**
→ You need to have text selected first — the command only appears when there is an active selection.

**Timeout error**
→ Increase `styleGuide.timeoutMs` in settings (e.g. to `300000` for slow Ollama models), or switch to a faster model like `mistral:7b`.

## Running Fully Offline with Ollama

Ollama runs the LLM entirely on your machine — no API key, no internet after the initial model download, no data leaves your computer.

### 1. Install Ollama

**macOS:** `brew install ollama`

**Ubuntu / WSL2:** `curl -fsSL https://ollama.com/install.sh | sh`

> WSL2: GPU acceleration works with an NVIDIA GPU + WSL2 driver. Without a GPU, Ollama runs on CPU — slower but functional.

### 2. Start Ollama

**macOS:** `ollama serve` or `brew services start ollama`

**Ubuntu:** `ollama serve` or `sudo systemctl start ollama`

**WSL2:** `ollama serve &`

### 3. Pull a model

| RAM | Model | Command |
|---|---|---|
| 8 GB | `mistral:7b` (fast, good formatting) | `ollama pull mistral:7b` |
| 8 GB | `llama3.1:8b` (alternative) | `ollama pull llama3.1:8b` |
| 16 GB | `mistral-small3.1:7b` (best instruction following) | `ollama pull mistral-small3.1:7b` |
| 32 GB+ | `llama3.3:70b` (highest quality) | `ollama pull llama3.3:70b` |

For style guide use (structured output, rule-following, citations) the **mistral** family produces cleaner results. Start with `mistral:7b`.

### 4. Configure `.env`

```bash
LLM_BACKEND=ollama
OLLAMA_MODEL=mistral:7b
OLLAMA_BASE_URL=http://localhost:11434
```

### 5. Test

```bash
make test
# Expected: Using backend: Ollama (mistral:7b) -> http://localhost:11434
```

**Notes:**
- First query loads the model into memory (~60–90s). Subsequent queries should be faster.
- Ollama must be running before `make server` or `make server-http`.
- If answers seem cut off, add `OLLAMA_NUM_CTX=8192` to `.env`.

## Generating Skills from Your PDF

`make skill` reads your PDF and generates **three different skill formats** in one shot — one for each environment you might work in:

```bash
make skill
```

Output:
```
skill_output/
├── system_prompt.md                    ← Claude.ai Project instructions
├── INSTRUCTIONS.md                     ← setup guide for the above
├── CLAUDE.md                           ← Claude Code instruction file
└── writing-style-guide-skill/
    └── SKILL.md                        ← reusable Claude Code skill
```

---

### Format 1 — Claude.ai Project

The easiest option — no server, no CLI, just claude.ai in your browser.

**Setup:**
1. Go to [claude.ai](https://claude.ai) → **Projects** → **Create project**
2. Click **Edit project instructions** → paste the contents of `skill_output/system_prompt.md`
3. Click **Add content** → upload your style guide PDF
4. Done — every conversation in the project is now style-guide-aware

**Example prompts:**
- *"Can I use contractions like 'can't' in documentation?"*
- *"Check this paragraph for style violations: [paste text]"*
- *"Rewrite this to comply with our style guide: [paste sentence]"*

---

### Format 2 — Claude Code `CLAUDE.md`

`CLAUDE.md` is a Markdown file that Claude Code reads automatically at the start of every session. Place it in your project root for project-scoped rules, or in `~/.claude/` to apply it globally across all projects.

```
Claude Code session starts
        │
        ▼
Reads CLAUDE.md (project root, then ~/.claude/)
        │
        ▼
Claude already knows your style rules before you type anything
```

**Project-level** (applies only when working in that project):
```bash
cp skill_output/CLAUDE.md <your-project-root>/CLAUDE.md
```

**Global** (applies to all Claude Code sessions):
```bash
cp skill_output/CLAUDE.md ~/.claude/CLAUDE.md
```

> If `~/.claude/CLAUDE.md` already exists, append instead of overwriting:
> ```bash
> cat skill_output/CLAUDE.md >> ~/.claude/CLAUDE.md
> ```

Once in place, Claude Code will automatically apply the style guide whenever you ask it to write, review, or edit documentation — no extra prompting needed.

---

### Format 3 — Claude Code Skill (`~/.claude/skills/`)

A skill is a reusable, named capability that Claude Code can explicitly load on demand. Unlike `CLAUDE.md` which is always loaded, a skill is consulted when the task matches its description — keeping context lean for sessions where style checking isn't needed.

```
Claude Code session
        │
        ▼
You ask: "Check this doc for style issues"
        │
        ▼
Claude sees skill description matches → loads SKILL.md into context
        │
        ▼
Applies style rules from the skill to your text
```

**Install:**
```bash
cp -r skill_output/writing-style-guide-skill ~/.claude/skills/
```

**Verify it's available** by starting a Claude Code session and asking:
```
What skills do you have available?
```

You should see `writing-style-guide` listed.

**Usage** — Claude Code will load it automatically when relevant, or you can reference it explicitly:
```
Use the writing-style-guide skill to review this paragraph: [paste text]
```

---

### Which format to use?

| Format | Works in | Always loaded? | Best for |
|---|---|---|---|
| **Claude.ai Project** | claude.ai browser | Per conversation | Non-developers, quick checks |
| **CLAUDE.md (project)** | Claude Code | Yes, in that project | Teams — shared in the repo |
| **CLAUDE.md (global)** | Claude Code | Yes, everywhere | Personal global style rules |
| **~/.claude/skills/** | Claude Code | No — on demand | Multiple style guides, lean context |
| **MCP server + extension** | Claude Desktop, VS Code | On demand | Developers, VS Code right-click checks |

> **Tip — combine them:** use the `~/.claude/skills/` skill for general Claude Code work, and the MCP server + VS Code extension for real-time checking while writing. They both read the same PDF and use the same RAG engine.

---

### Keeping skills up to date

Re-run `make skill` whenever the PDF changes — it regenerates all three formats at once. Then copy the updated files to wherever you installed them.

## Switching Backends

Change one line in `.env` and restart the server:

```bash
LLM_BACKEND=claude       # Anthropic Claude API
LLM_BACKEND=copilot      # GitHub Copilot (OpenAI-compatible)
LLM_BACKEND=openrouter   # OpenRouter (free & paid models)
LLM_BACKEND=ollama       # Fully local — nothing leaves your machine
```

## Tuning

| `.env` variable | Default | Effect |
|---|---|---|
| `RETRIEVER_K` | `5` | PDF chunks retrieved per query. Higher = broader context. |
| `CHUNK_SIZE` | `1000` | Characters per chunk. Smaller = more precise retrieval. |
| `CHUNK_OVERLAP` | `200` | Overlap between chunks — prevents context being cut at boundaries. |
| `EXTRA_CONTEXT` | _(empty)_ | Business unit specialisation injected into prompts, e.g. `"You are reviewing content for a financial services audience."` |

## Troubleshooting

**PDF not found**
→ Use an absolute path: `PDF_PATH=<PATH>/style-guide.pdf`

**`ModuleNotFoundError` on first run**
→ Use `make test` — it always calls `.venv/bin/python`. If running manually: `.venv/bin/python rag.py`

**Packages installed but not found (multiple Python versions)**
→ `make reinstall` — recreates the venv cleanly.

**Connecting to wrong host / `ANTHROPIC_BASE_URL` ignored**
→ Add `ANTHROPIC_BASE_URL=https://api.anthropic.com` to `.env`. The code uses `load_dotenv(override=True)` so `.env` always wins over shell environment variables.

**`GITHUB_TOKEN` not set / Copilot 401**
→ Generate a token at github.com/settings/tokens with the **Models** read scope.

**Ollama: connection refused**
→ Run `ollama serve`. On WSL2: `ollama serve &` in your WSL terminal.

**Ollama: answers cut off**
→ Add `OLLAMA_NUM_CTX=8192` to `.env` and restart.

**First query is slow**
→ Normal — PDF is embedded on first use. Fast from second query onward.

**Claude Desktop: no 🔧 icon**
→ Python path in config must point to `.venv/bin/python`. On WSL2 use `wsl.exe` as the command.

**VS Code extension: Server Offline**
→ `make server-http` must be running in a separate terminal.

**VS Code extension: Check Selection greyed out**
→ Select text first — the command only appears with an active selection.

**WSL2: `make` not found**
→ `sudo apt install -y make`

**WSL2: slow performance**
→ Keep files in `/home/...` not `/mnt/c/...`.
